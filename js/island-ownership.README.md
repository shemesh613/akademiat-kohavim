# island-ownership.js — מדריך שילוב למתאם

מנגנון הבעלות של האי, לפי ההמלצה המכריעה ב-`RESEARCH_OWNERSHIP.md`:
- **שטח משותף = 100% אוטומטי** — סדר עלילתי + מיקום קבועים מראש בקוד לכל 80 המבנים.
  המערכת בונה לבד כשהקופה חוצה את הסף, עם קונפטי וטוסט. אף אדם לא בוחר — אין את מי להאשים.
- **חלקה אישית = 100% בחירת הילד** — מטבע אישי (`student.island.personalCoins`) שנצבר
  מאותן נקודות (10 נק' = 1 אבן כיתתית **וגם** 1 מטבע אישי, בלי פיצול), קטלוג של עד 4 פריטים
  גלויים, בלי אישור מורה, מטבעות לא פוקעים.
- **הגנות** — בנייה סופית לכולם (אין מחיקה), אין השוואות בין תלמידים, ניסוח חיובי בלעדי.

**חשוב: הקובץ לא עורך את `island-engine.js` בשום צורה.** האכיפה נעשית ע"י החלפת
`window.Island.place` / `window.Island.remove` בעטיפות בתוך `init()` (ההפניות המקוריות
נשמרות פנימית ומשמשות את הבנייה האוטומטית), ובנוסף שכבת CSS שמסתירה את פלטת הבנייה
הידנית (`.ak-isl-shop`), בורר החלקות (`.ak-isl-plots`) וכפתור המחיקה (`.ak-isl-item.del`)
של המנוע — כי ה-HUD הפנימי קורא ישירות ל-`placeAt`/`removeAt` ועוקף כל עטיפה. ההסתרה
הזו היא חלק מהאבטחה, לא קישוט.

---

## 1. תג ה-script ל-index.html

להוסיף **אחרי** `ak-bridge.js` ו-`island-engine.js` (חובה אחרי שניהם — המודול עוטף את
`window.Island` שהמנוע יוצר):

```html
<script src="js/ak-bridge.js"></script>
<script src="js/island-content.js"></script>
<script src="js/island-engine.js"></script>
<script src="js/island-ownership.js"></script>   <!-- ← חדש -->
<script src="js/class-games.js"></script>
```

(אם מסיבה כלשהי הוא נטען לפני המנוע — הוא ימתין לבד ל-`window.Island` עד 30 שניות ויעטוף
ברגע שיופיע. אבל הסדר למעלה עדיף.)

## 2. קריאת `init()` — איפה

פעם אחת, מיד אחרי טעינת הסקריפטים (או ב-`DOMContentLoaded` / init הכללי של המתאם):

```html
<script>
  if (window.IslandOwnership) IslandOwnership.init();
</script>
```

`init()` אידמפוטנטי (בטוח לקרוא פעמיים). הוא: עוטף את `Island.place`/`remove`, מזריק את
ה-CSS המגן, ומפעיל לולאה פנימית (כל 1.5 שנ') שמסנכרנת את ה-UI (טיזר + פאנל תלמיד)
**וגם קוראת בעצמה ל-`checkAutoBuild` כל ~4.5 שניות** — ראו סעיף 4.

## 3. חיבור `onPointsAdded` בתוך `addPoints`

ב-`index.html`, בתוך `function addPoints(studentId, amount, fromX, fromY)` (שורה ~2553),
**לפני** קריאת ה-`save()` הקיימת (ולצד ה-hook של `ClassGames.onPointsAdded`):

```js
if (window.IslandOwnership) {
  try { IslandOwnership.onPointsAdded(studentId, amount); } catch (e) {}
}
```

- מקבל `studentId` (מחרוזת) או אובייקט תלמיד — שניהם עובדים.
- סכומים שליליים (כפתור ‎-10) **נבלעים בכוונה** — הארנק האישי לעולם לא נענש (מחקר: ניסוח חיובי בלעדי).
- שארית נקודות (<10) נשמרת ב-`student.island.pointsRemainder` ומצטברת.
- **הפונקציה לא שומרת בעצמה** — היא מסתמכת על ה-`save()` שכבר קיים בסוף `addPoints`.
  לכן חובה למקם את הקריאה לפניו.

## 4. `checkAutoBuild` — interval

לא חובה לחבר כלום: הלולאה הפנימית של `init()` כבר קוראת `checkAutoBuild(null)` כל ~4.5
שניות. אם המתאם מעדיף שליטה מפורשת (למשל לצד בדיקות אחרות):

```js
setInterval(function () {
  if (window.IslandOwnership) IslandOwnership.checkAutoBuild(AK.getActiveClass());
}, 5000);
```

בטוח לחלוטין לקרוא משני המקומות — יש cooldown פנימי של 25 שניות בין שתי בניות
(רגעי גילוי מרווחים ודרמטיים, לא הצפה), וכל קריאה בונה לכל היותר מבנה אחד.

הערות התנהגות:
- בונה רק כשהמנוע כבר אותחל (`Island.open` נקרא לפחות פעם אחת) — אם האי סגור כשנחצה
  סף, המבנה ייבנה ברגע שהאי ייפתח (גילוי על המקרן — זה אפילו רצוי).
- "הבא בתור" נגזר דטרמיניסטית מ-`klass.island.items` (אין שדה state חדש ברמת הכיתה) —
  עמיד לרענון, לסנכרון ענן ול-state ישן.
- בין אזורים פתוחים נבנה תמיד באזור עם הכי מעט מבנים מהתוכנית (שוויון → סדר קבוע) —
  אזור שנפתח זה עתה מתמלא מהר ואף אזור לא נשאר ריק.
- אם המשבצת המתוכננת תפוסה (בנייה ידנית ישנה) — חיפוש ספירלי לתא פנוי סמוך.

## 5. ה-API המלא

```js
window.IslandOwnership = {
  init(),                          // חד-פעמי; עוטף את Island.place/remove + מפעיל לולאה
  canBuild(regionKey, studentId),  // מורה: הכל; תלמיד: רק 'plot_<שלו>'; משותף: אף תלמיד
  onPointsAdded(studentOrId, amt), // 10 נק' = +1 מטבע אישי; שליליים נבלעים; לא שומר
  checkAutoBuild(klass),           // בונה את הבא בתור אם הקופה חצתה סף (klass=null → פעילה)
  teaser(klass),                   // {pct:0..1, ready:Boolean, done:Boolean} — בלי לחשוף מה נבנה
  personalCatalog(studentOrId),    // עד 4 פריטים: [{id,em,n,cost,pcost}] (pcost=מטבעות אישיים)
  // עזרים:
  buildPersonal(studentOrId, itemId), // בונה בחלקה של התלמיד בתא הפנוי הבא (מהמרכז החוצה)
  getPersonalCoins(sid),           // יתרת מטבעות אישיים
  setActiveStudent(sid),           // דריסת זיהוי התלמיד הפעיל (ברירת מחדל: studentSelectedId)
  setManualUI(true|false),         // דיבוג בלבד: מחזיר את פלטת הבנייה הידנית של המנוע
};
```

## 6. מה קורה מאחורי הקלעים — דברים שחשוב שהמתאם יידע

1. **תשלום בחלקה אישית**: `placeAt` של המנוע מחייב תמיד את קופת הכיתה. העטיפה מצלמת
   snapshot של `coins`/`spent`, מפקידה זמנית רק את החוסר (אם יש), נותנת למנוע לבצע,
   ומשחזרת את ה-snapshot במדויק + מחשבת `level` מחדש. נטו: בנייה אישית לא נוגעת בקופה
   הכיתתית, בספי האזורים או ב-level. הפקדה שהייתה חוצה בטעות סף אזור נעול — נדחית בעדינות
   (מקרה קצה נדיר; הספים משוכפלים מ-SPEC פרק 3 — אם ישתנו שם, לעדכן גם ב-`REGION_THRESHOLDS`).
2. **מחיר אישי**: `pcost = max(1, ceil(cost/10))` — הפריט הראשון בכל אזור עולה 1 מטבע
   (= 10 נקודות), כדי שכל ילד יגיע לפריט ראשון תוך שבוע-שבועיים (הגנת "חלקה ריקה").
3. **מחיקה חסומה לכולם** — כולל מורה, כולל במצב `setManualUI(true)` (כפתור המחיקה מוסתר
   תמיד ו-`Island.remove` תמיד מחזיר false עם הודעה חיובית). בנייה סופית = אין ריבים.
4. **זיהוי תלמיד במצב בית**: קורא את `studentSelectedId` (משתנה let גלובלי ב-index.html)
   דרך binding לקסיקלי. אם המתאם ישנה את המנגנון — לקרוא `setActiveStudent(sid)` מתוך
   `enterStudentMode` ו-`setActiveStudent(null)` ביציאה.
5. **UI שהמודול מוסיף לתוך מכולת האי** (רק כשה-HUD של המנוע קיים):
   - `.ak-own-teaser` — "✨ מתקרבים לגילוי הבא! ❓ NN%" מ-70% ומעלה. לא חושף מה המבנה.
   - `.ak-own-panel` — פאנל אישי לתלמיד במצב בית: יתרה, עד 4 פריטים, בנייה בלחיצה.
     המיקום בחלקה נבחר אוטומטית מהמרכז החוצה (גם פריט אחד נראה "מלא" — הגנת שיימינג).
     חלקה בלי פריטים מוצגת "האדמה מוכנה לצמוח!" — לעולם לא "ריקה".
   - המודול לעולם לא מציג נתונים של תלמיד אחד לתלמיד אחר, ואין שום דירוג/השוואה.
6. **state חדש** (עם fallback מלא, לא שובר מיגרציות):
   - `student.island = { personalCoins, pointsRemainder, spentCoins }` — חדש, ברמת תלמיד.
   - ברמת הכיתה — **אין שדות חדשים** (הבא-בתור נגזר מ-`items`).
7. **מבנים אוטומטיים נרשמים עם `by:null`** — אין שם של ילד על מבנה משותף, אין את מי להאשים.
