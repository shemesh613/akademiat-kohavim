# island-life.js — מדריך שילוב למתאם

מודול נפרד לגמרי, לא נוגע ב-`island-engine.js`. מזריק חיים לאי: 24 התלמידים כדמויות
תלת-מימדיות אמיתיות (דרך `window.build3DCharacter` הקיים), חיית-כיתה + עץ-לכל-אזור שגדלים
לפי כלכלת אבני-הבניין, מזג אוויר/עונות, יום-לילה אמביינטי, חגי ישראל, גילוי וסוד, והאי מגיב
לפעילות הכיתה. אם הקובץ חסר — האי ממשיך לעבוד בדיוק כמו היום, ללא שום שינוי.

## 1. מה להוסיף ל-`index.html`

שורת `<script>` אחת, **אחרי** `island-engine.js` (הסדר לא קריטי טכנית כי הטעינה
א-סינכרונית ביחס להרצה, אבל זה הסדר ההגיוני):

```html
<script src="js/island-content.js"></script>
<script src="js/island-engine.js"></script>
<script src="js/island-life.js"></script>   <!-- הקובץ הזה -->
```

## 2. שלוש שורות הקריאה שצריך להוסיף ל-`js/island-engine.js`

**אל תשנו שום דבר אחר בקובץ.** שלוש הוספות בלבד, כל אחת שורה אחת, בתוך ה-IIFE הקיים
(כדי ש-`ISL` הפנימי יהיה בסקופ). כל קריאה עטופה ב-`typeof` כדי שלא לקרוס אם
`island-life.js` לא נטען (מצב הדגמה/פיתוח).

### 2.1 `detach()` — בתחילת `initScene(container)` (סעיף 15 בקובץ)

רשת ביטחון בלבד — הפונקציה הזו רצה כיום פעם אחת בחיי הדף, אז זה בעיקר להגנה על
עתיד שבו `initScene` ירוץ שוב. מוסיפים **מיד אחרי** השורה הראשונה של הפונקציה:

```js
function initScene(container) {
  if (window.IslandLife && typeof window.IslandLife.detach === 'function') window.IslandLife.detach();  // <-- חדש
  ISL.container = container;
  var canvas = document.createElement('canvas');
  ...
```

### 2.2 `attach(ISL)` — בסוף `initScene(container)`, אחרי `resizeNow();`

זה הרגע שבו `ISL.scene`/`ISL.camera`/`ISL.container` כבר קיימים ותקינים:

```js
function initScene(container) {
  ...
  setupInput(canvas);
  buildHud(container);
  setupResize(container);
  resizeNow();
  if (window.IslandLife && typeof window.IslandLife.attach === 'function') window.IslandLife.attach(ISL);  // <-- חדש
}
```

### 2.3 `tick(t, dt)` — בסוף `stepFrame()`, **ממש לפני** `renderer.render(...)`

חשוב שזה יקרה **אחרי** `updateDayNight(t, ISL.scene)` באותה פונקציה — כך שהעדכונים
העדינים של island-life.js לצבע ה-fog (מצב-רוח/גשם/ערפל) מוחלים על גבי הבסיס שהמנוע כבר
חישב לאותו פריים, ולא נדרסים על ידו:

```js
function stepFrame() {
  ...
  updateHud(t);
  if (window.IslandLife && typeof window.IslandLife.tick === 'function') window.IslandLife.tick(t, ISL.clock.elapsed > 0 ? dt : 0.016);  // <-- חדש
  ISL.renderer.render(ISL.scene, ISL.camera);
}
```

בפועל `dt` הקיים בפונקציה כבר תקין תמיד (מחושב כמה שורות למעלה ב-`stepFrame`) —
אפשר גם פשוט `window.IslandLife.tick(t, dt);`. הניסוח למעלה הוא רשת ביטחון בלבד
למקרה קצה של `dt` לא תקין בפריים הראשון; לא חובה.

**זהו. שלוש שורות, שלוש נקודות הזרקה.** שום דבר אחר ב-`island-engine.js` לא זז.

## 3. מה קורה אם `island-life.js` לא נטען

`window.IslandLife` לא קיים → כל שלוש הבדיקות `typeof window.IslandLife...` נכשלות
בשקט → האי מתפקד בדיוק כמו היום, בלי אף שגיאה בקונסולה.

## 4. חוזה ה-API שהמודול חושף

```js
window.IslandLife = {
  attach(ISL),           // חד-פעמי, נקרא מתוך initScene()
  tick(t, dt),            // כל פריים, נקרא מתוך stepFrame()
  detach(),                // רשת ביטחון, נקרא מתחילת initScene()
  setReducedMotion(bool)  // אופציונלי — אם רוצים toggle ידני נוסף על prefers-reduced-motion
};
```

`attach(ISL)` מקבל **הפניה ישירה** (לא עותק) לאובייקט ה-`ISL` הפנימי של המנוע —
`scene`/`camera`/`renderer`/`regionGroups`/`regionTier`/`activeId`/`clock`/`ambient`/`container`.
המודול קורא משם בלבד ומוסיף אובייקטים עצמאיים ל-`ISL.scene` (קבוצת-שורש
`ak-island-life` + חיית הכיתה + עצי האזורים + מזג אוויר וכו') — **לא נוגע** ב-
`regionGroups`/`regionTier` וב-state הפנימי האחר של המנוע.

## 5. מה נשמר ל-state (`klass.island`)

בנוסף לשדות הקיימים (`coins`/`spent`/`regions`/`items`/`level`/`history`), המודול
מוסיף ומנהל בעצמו (עם fallback מלא, לא שובר כיתות ישנות):

```js
klass.island.pet = {
  stage: 0,        // 0=ביצה, 1..8 = מס' אזורים פתוחים כשהיא בקעה (אבן-דרך גדולה לכל אזור)
  fine: 0,         // 0..8 — קפיצות קטנות בתוך ה"עידן" הנוכחי (כל 3-5 ימי לימוד בערך)
  lastGrow: 0,     // timestamp של קפיצת הגדילה האחרונה
  born: 0,         // timestamp של בקיעת הביצה
  monthMark: { stage: 0, t: 0 }   // "כאן היינו לפני חודש" בסרגל המדידה
};
klass.island.tree = {
  // מפתח = מזהה אזור (beach/forest/farm/village/mountain/desert/volcano/sky).
  // כל אזור פתוח מקבל רשומה משלו — עץ נפרד עם מסלול גדילה עצמאי (~חודש, לפי
  // % התקדמות בתוך האזור, לא תאריך). אזור שכבר "עבר" (נפתח אזור מתקדם ממנו)
  // קופא בשלב הבשל/פורח לתמיד — עדות חיה לציר הזמן של השנה.
  beach: { stage: 0, fine: 0, lastGrow: 0, monthMark: { stage: 0, t: 0 } }
  // ... נוצר lazily לכל אזור פתוח
};
```

שני האובייקטים נשמרים אוטומטית דרך `AK.save()` (אותו hook הקיים ב-window.AK) בכל
פעם שיש קפיצת גדילה — אין צורך בשום פעולה נוספת מהמתאם.

## 6. החלטות עיצוב שכדאי לדעת

1. **גדילת חיית הכיתה/עצים לפי כלכלה, לא תאריך** — `stage`/`fine` נגזרים
   מ-`isl.regions.length` ומ-`(coins+spent)` מול `REGION_THRESHOLDS` הקבועים
   (0,120,300,520,780,1050,1350,1700 — שוכפלו כאן במכוון, זהים לאלה שב-
   `island-engine.js`; אלה "מקור אמת" קבוע לפי ה-README של סוכן A). כיתה מהירה
   לא "מפספסת" שלבים וכיתה איטית לא נענשת — היצור פשוט נח/ישן, אף פעם לא עצוב.
2. **חגי ישראל הם היוצא מן הכלל היחיד שכן פועל לפי תאריך לועזי אמיתי** — טבלה
   קשיחה ל-2026–2029 (ר' סעיף 1 בקוד). 2026–2028 מבוססים על חיפוש מאומת
   (Chabad/Hebcal); 2029 נגזר מקירוב אורך-חודש סטנדרטי עם טווח מורחב לביטחון.
   **יש לוודא/לעדכן מדי שנה** מול לוח שנה עברי רשמי — התאריכים מסומנים בקוד.
3. **דמויות התלמידים נבנות דרך `window.build3DCharacter` הקיים**, לא שוכפלו —
   אותו כובע/משקפיים/אאורה/חיית מחמד בדיוק כמו בחנות. build3DCharacter מנקה את
   ה-`scene` שהוא מקבל ומזריק אליה תאורה משלו, אז המודול קורא לו על סצנה זמנית
   נפרדת (`tempCharScene`) ומוציא ממנה רק את גוף הדמות + חיית המחמד (התאורה
   נזרקת — לא רוצים 5 אורות נוספים לכל תלמיד בסצנת האי המשותפת).
4. **LOD תלמידים: 6 מלאים + impostor לשאר** — 6 הקרובים ביותר למצלמה (מבין אלו
   שהאזור-בית שלהם כרגע ב-`regionTier==='full'`) מקבלים rig מלא (~40 meshes,
   דרך build3DCharacter). כל השאר מקבלים Sprite זול (תג-עיגול צבעוני + אות
   ראשונה של השם). קאש LRU של עד 10 rigs בזיכרון, בנייה חדשה מוגבלת לאחת
   לפריים (לא לגמגם). המשימה אפשרה 6-8 — נבחר 6 לבטיחות ביצועים.
5. **חיית מחמד עוקבת בעיכוב טבעי (טרייל)** — מיוצאת כ-`Group` **נפרד** מגוף
   הדמות (לא מקונן בפנים), מצורפת ישירות ל-root group של המודול ולא ל-wrapper
   המסתובב של התלמיד/ה — כדי שמיקומה תמיד יהיה בקואורדינטות עולם פשוטות (לא
   צריך להתמודד עם ה-scale/offset הפנימיים של נירמול גוף הדמות).
6. **הרגליים על y=0 מנורמלות דרך bounding-box**, לא offset קשיח — עובד גם ל-
   דמויות GLB (Xbot/Soldier/Robot) וגם לדמויות פרוצדורליות, עם קלאמפ שפוי כדי
   שאאורות מהבהבות (אש/קרח/טבעת-קדושה) שמתפשטות מתחת לרגליים לא "יתפסו" כרצפה.
7. **חוקי תוכן**: בלי שם ה' בשום טקסט, בלי דמויות חרדיות (כובע פורים/מסכה —
   צבעוני וצנוע, לא שטריימל/קפוטה), הודעות בקבוק-המסר תמיד קולקטיביות-חיוביות
   (בלי "מי תרם הכי מעט" — RESEARCH.md פרק 1.4/10.3).

## 7. תקציב ביצועים (מדוד/מוערך — ראו גם README §5 של island-engine.js)

יעד: 60fps על טאבלט, בהתאם ל-SPEC §4/§5.2. הנחיית המשימה הייתה "השתמש ב-
InstancedMesh לכל דבר חוזר" — יושם לגשם, עלים, פרפרים, קישוטי חג, נצנצי חיית-כיתה
מפוארת, וכיפת עלווה של כל עץ. הערכת draw-calls נוספים **על גבי** אלה שכבר קיימים
ב-island-engine.js (קרקע אזורים + פריטים + ים + שמיים + HUD, שם כבר יש תקציב-על
של 15,000 משולשים לאזור פעיל):

| רכיב | draw calls | הערות |
|---|---:|---|
| 6 תלמידים ב-LOD מלא | ~234 | ~39 meshes/דמות (build3DCharacter פרוצדורלי) × 6 |
| 18 תלמידים ב-impostor | 18 | Sprite בודד לכל אחד |
| 24 שלטי-שם | 24 | Sprite לכל תלמיד/ה (גם מלא וגם impostor) |
| חיית הכיתה | 8–12 | תלוי בשלב (glow/כתר/נצנצים ב-InstancedMesh משלב 8) |
| עצים (עד 3 גלויים בו-זמנית — רק אזורים ב-'full') | ~36 | גזע+קליפה+2×InstancedMesh (עלווה/פרי) × 3 |
| סרגלי מדידה (חיה + עד 3 עצים) | ~30 | עמוד+סימונים+2 sprites לכל אחד |
| גשם/עלים/פרפרים | עד 3 | InstancedMesh בודד לכל תופעה, פעיל רק כשרלוונטי |
| קשת אחרי גשם | 6 | 6 קשתות-טורוס חלקיות, רק ~22 שניות אחרי גשם |
| כוכבים/ירח/מדורה/פנסים | ~5 | רק במצב `ambient===true` ובלילה |
| חגים | 0–14 | 0 ברוב השנה; עד ~14 draws בשיא (למשל חנוכייה+לולב בו-זמנית) |
| גילוי (ספינה/כוכב-נופל/בקבוק/פינות נסתרות) | 0–8 | רוב הזמן 0 — מופיעים לרגעים בודדים |
| **סה"כ (מצב טיפוסי, לא שיא)** | **~370–420** | בעיקר תלמידים; שאר הרכיבים <100 ביחד |

**נקודת כאב עיקרית**: LOD המלא של תלמידים (~39 meshes/דמות, כי `build3DCharacter`
עשיר בפירוט — קפסולות בנות 3 meshes לכל גפה, עיניים עם ברק וכו'). אם פרופיילינג
בפועל (במכשיר יעד אמיתי) יראה בעיה — הכי קל להוריד את `MAX_FULL_VILLAGERS`
(קבוע בראש סעיף 5 בקוד, כרגע 6) ל-4, שחוסך ~78 draw calls לכל תלמיד שיורד.
`prefers-reduced-motion` (או `IslandLife.setReducedMotion(true)`) לא מוריד draw
calls (הן כבר קיימות בסצנה) אלא מכבה אנימציה/תזוזה — לביצועים חוסכים מ-LOD/קאש.
