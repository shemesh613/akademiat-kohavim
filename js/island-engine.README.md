# island-engine.js — מדריך שילוב למתאם

קובץ יחיד, עצמאי, ES5-friendly. לא נוגע ב-DOM חיצוני מלבד המכולה שמקבל וה-`<style>`
שהוא מזריק לעצמו (`#ak-island-style`). תלוי ב-`window.AK` (אופציונלי בזמן ריצה, אבל
צריך להיות קיים בפועל כדי לשמור state) וב-`window.IslandContent` (אופציונלי — אם
חסר, מוצגים placeholders + אזהרת קונסול, בלי קריסה).

## 1. מה להוסיף ל-`index.html`

הוספה של שתי שורות `<script>`, **אחרי** Three.js ו-**אחרי** ש-`window.AK` כבר
מוגדר, ולפני שקוראים ל-`Island.open(...)`:

```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/three.js/r128/three.min.js"></script>
<!-- ... window.AK = {...} מוגדר כאן ע"י המתאם ... -->

<script src="js/island-content.js"></script>   <!-- סוכן B — אופציונלי, אפשר בכל סדר -->
<script src="js/island-engine.js"></script>     <!-- הקובץ הזה -->
```

**הערה:** סדר הטעינה בין `island-content.js` ל-`island-engine.js` לא קריטי —
המנוע בודק אם `window.IslandContent` קיים ב-`open()`, ואם לא, בודק שוב כל שנייה
במשך 20 שניות ומרענן אוטומטית ברגע שהוא מופיע. אין צורך לתאם סדר טעינה.

## 2. איך פותחים/סוגרים את האי

```js
// לפתוח (בלחיצת "האי שלנו" למשל):
Island.open(document.getElementById('island-container'));

// לפתוח כמסך אמביינט ברקע (סעיף 5.2 ב-SPEC — "מסך רקע"):
Island.open(document.getElementById('island-container'), { ambient: true });
// או בכל רגע:
Island.setAmbient(true);   // מצב "סיור אוטומטי" קולנועי
Island.setAmbient(false);  // חוזר לשליטה ידנית

// לסגור (לא הורס את הסצנה — פתיחה חוזרת מיידית וזולה):
Island.close();
```

- `container` יכול להיות אלמנט DOM או מחרוזת id. אם לא מועבר כלל, המנוע יוצר
  ולהוסיף `<div id="ak-island-root">` שממלא את כל המסך (`position:fixed;inset:0`).
- המנוע מנהל לולאת רינדור עצמאית (`requestAnimationFrame`) — **אין צורך לקרוא
  ל-`Island.tick()` בעצמכם**. הפונקציה חשופה למקרה שתרצו לדחוף פריים ידני (למשל
  מיד אחרי פעולה חשובה), אך זה אופציונלי לגמרי.
- `Island.close()` עוצר את הלולאה בלבד; הסצנה/מטמון נשארים "חמים" בזיכרון כדי
  שפתיחה הבאה תהיה מיידית (מתאים לשימוש יומיומי חוזר).

## 3. שאר ה-API הציבורי

```js
Island.unlockRegion('forest');           // פתיחה ידנית (חוץ מהאוטומטית לפי כלכלה)
Island.place('beach', 'palm', 3, 2);     // הצבת פריט: (regionId, itemId, x, z [,rot][,studentId])
Island.remove('beach', 3, 2);            // הסרה: (regionId, x, z) — מחזיר חצי מחיר
Island.focusRegion('farm');              // מעביר את המצלמה/HUD לאזור אחר (עם lerp חלק)
```

כל הפעולות האלה כבר קוראות ל-`AK.save()` בעצמן — אין צורך לשמור ידנית אחריהן.

**פתיחת אזורים אוטומטית:** המנוע בודק כל 2 שניות (וגם ב-`open()`) האם
`coins+spent` של הכיתה חצה סף של אזור נעול, ואם כן פותח אותו לבד — אנימציה,
קונפטי, סאונד, טוסט. **אין צורך שהמתאם/המורה יקראו לזה בעצמם** — זה מימוש
פרק 5.1 ב-SPEC (אפס עומס למורה).

## 4. מבנה ה-DOM שנוצר

`Island.open(container)` מוסיף לתוך `container`:
- `<canvas>` ממלא 100%/100% (הרנדר של Three.js).
- שכבת HUD (`<div class="ak-isl-hud">`) עם: מונה אבנים, שם אזור, פס התקדמות
  לאזור הבא, פס בחירת "בונים במשותף / חלקה אישית של תלמיד/ה", פלטת פריטים
  לבנייה/הסרה. כל הטקסטים ≥20-48px לפי דרישת המקרן (SPEC §5.2).
- `container` מקבל `position:relative` אוטומטית אם לא הוגדר.

אין תלות ב-CSS חיצוני — כל הסגנון מוזרק פנימית (`<style id="ak-island-style">`).

## 5. מה סוכן B (`island-content.js`) צריך לספק — `window.IslandContent`

```js
window.IslandContent = {
  // מערך של עד 8 אובייקטים — אחד לכל id קבוע (הסדר לא משנה, המנוע ממיין לפי id):
  // 'beach' | 'forest' | 'farm' | 'village' | 'mountain' | 'desert' | 'volcano' | 'sky'
  REGIONS: [
    {
      id: 'beach',                 // חובה — אחד מ-8 המזהים הקבועים למעלה
      name: 'חוף הכוכבים',          // אופציונלי — למנוע כבר יש שם ברירת מחדל זהה מה-SPEC
      theme: { sky:0x8fd8ff, fog:0xbfe8ff, ground:0xe6cf94, accent:0xf5e7ae }, // אופציונלי, hex numbers
      items: [                     // קטלוג הפריטים הניתנים לבנייה *באזור הזה* (גם בחלקות האישיות שבתוכו)
        { id:'palm', em:'🌴', n:'דקל', cost:20 },
        { id:'flower', em:'🌸', n:'פרח', cost:8 }
        // ... ~10 פריטים לאזור בממוצע, מחיר ב"אבני בנייה" (יעד: ~1,800/שנה סה"כ לכל 8 האזורים)
      ]
    },
    // ... עוד 7 אזורים
  ],

  // builder לכל item id שמופיע בקטלוג של איזשהו אזור (המפתח = it.id):
  BUILDERS: {
    palm: function () {
      var g = new THREE.Group();
      // ... 8-25 meshes, לפחות 3 צבעים (לפי SPEC §4) ...

      // אופציונלי: פרט מונפש אחד. המנוע קורא לזה כל פריים (t = שניות מאז הפתיחה, dt = דלתא):
      g.userData.animate = function (t, dt) {
        // לדוגמה: mesh.rotation.z = t * 1.6;
      };
      return g; // Group — לא Mesh בודד
    }
    // ... ~80 builders
  }
};
```

**חוקים חשובים לסוכן B:**
- `BUILDERS[id]()` חייבת להחזיר `THREE.Group` חדש בכל קריאה (לא לשתף מופע יחיד בין
  קריאות — המנוע קורא לזה מחדש בכל rebuild של האזור, וממקם/מסובב את התוצאה בעצמו
  דרך `.position`/`.rotation.y`. **אל תגדירו position/rotation קבועים בשורש
  ה-group עצמו** — הבנייה תדרוס אותם).
- אם `BUILDERS[id]` זורק שגיאה או לא מוגדר — המנוע לוכד את זה, מציג קובייה+חרוט
  placeholder עם אימוג'י הפריט (אם יש בקטלוג) ורושם `console.warn` **פעם אחת**
  לכל id חסר (לא מציף את הקונסולה).
- שדה `cost` הוא ב"אבני בנייה" (`klass.island.coins`), לא נקודות תלמיד.
- `theme` הוא אופציונלי לגמרי — יש ברירת מחדל צבעונית לכל 8 הביומות כבר במנוע.
  שימוש ב-`theme` דורס רק את הצבע הבסיסי של הקרקע/ערפל, לא משפיע על שום דבר אחר.
- **סף הפתיחה של כל אזור (`threshold`) הוא קבוע במנוע** לפי טבלת הכלכלה בפרק 3
  של ה-SPEC ואי אפשר לדרוס אותו דרך `IslandContent` — זה נשאר "מקור אמת" יחיד
  כדי לא לשבור את חוזה הכלכלה מול class-games.js וסוכנים אחרים.

## 6. החלטות עיצוב שכדאי לדעת (חריגות/פרשנויות מה-SPEC)

1. **מפת האזורים** מסודרת על טבעת-ספירלה אמיתית (רדיוס גדל ב-7.5 עבור כל אזור
   נוסף, זווית קבועה של 45°/אזור) — לא רק "עיגול שטוח". אזורים מחוברים בגשרי-עץ
   פשוטים כשגם השכן וגם הוא עצמו פתוחים, לחיזוק תחושת "ארכיפלג" אחד.
2. **חלקה אישית (SPEC §6)** ממומשת כפלטפורמת-לוויין 3x3 המחוברת בגשר לאזור
   הפעיל, מסודרת בטבעת סביב האי (לא כגזרה מתוך רשת ה-14x14 המשותפת) — כך
   שהמנגנון עובד לכל גודל כיתה (גם 30+ תלמידים) בלי להתנגש עם שטח הבנייה
   המשותף. הבנייה בחלקה האישית מופעלת דרך שורת "צ'יפים" ב-HUD ("בונים במשותף"
   מול "⭐ שם תלמיד/ה") — זו הדרך שבה מפעיל המקרן/ועדת האי בוחר איפה לבנות.
3. **`level` (1..40):** ה-SPEC מגדיר את השדה אך לא נוסחה. המנוע מחשב
   `level = clamp(1 + floor((coins+spent)/45), 1, 40)` כברירת מחדל סבירה שמגיעה
   ל-40 בערך בסביבות ה-1,755 אבנים (תואם את יעד סוף השנה בפרק 3). אפשר לשנות
   בקלות בפונקציה `recalcLevel()` אם רוצים נוסחה אחרת.
4. **ים/שמיים/מחזור יום-לילה:** גל על גיאומטריה (CPU, לא vertex shader) —
   נבחר לפשטות ויציבות על פני shader מותאם-אישית; התקציב זול (44×44 vertices).
   מחזור יום/לילה פעיל רק במצב `setAmbient(true)` (כדי לא "להחשיך" שיעור פעיל
   על המקרן); במצב רגיל השמיים/אור קבועים על "יום בהיר".
5. **ביצועים:** שלוש רמות פירוט לפי מרחק מהאזור הפעיל — `locked` (ערפל+שלט),
   `lod` (בליטת קרקע צבעונית בלי פריטים) לאזורים פתוחים לא-שכנים, `full` (קרקע
   + כל הפריטים + עשב/סלעים ב-InstancedMesh + חלקות אישיות) לאזור הפעיל
   ושכניו בטבעת. רק אזורים ב-`full`/`lod`/`locked` נמצאים בכלל ב-scene graph —
   שאר האזורים לא נטענים לזיכרון GPU כל עוד אינם רלוונטיים.
6. **קטלוג פריטים לחלקה אישית = קטלוג האזור המארח.** לא בנינו קטלוג נפרד קטן
   לחלקות אישיות — פשטות, ותחושת "אותה חנות, שני מקומות לבנות בהם".

## 7. מה קורה אם `IslandContent` חסר לגמרי (מצב הדגמה/פיתוח)

המנוע ממשיך לעבוד במלואו: קטלוג גיבוי פנימי בן 6 פריטים כלליים (`FALLBACK_ITEMS`),
ו-placeholder-ים צבעוניים (קובייה+חרוט+אימוג'י) לכל item id. אזהרת קונסולה אחת
לכל item/region חסר (`[Island] ...`), בלי לקרוס ובלי להציף לוג.
